
1. Какие основные принципы REST архитектуры вы знаете?

Ресурсо-ориентированность - все сущности представляются как ресурсы с уникальными URI
Единообразие интерфейса - использование стандартных HTTP-методов (GET, POST, PUT, DELETE, PATCH)
Бессостояние - каждый запрос содержит всю необходимую информацию для его обработки
Кэшируемость - ответы могут быть закэшированы для улучшения производительности
Слоистая система - архитектура может состоять из нескольких слоев
Код передаваемости по требованию - клиент может получать код для выполнения

2. В чем разница между HTTP методами PUT и PATCH?

PUT - используется для полного обновления ресурса. Клиент должен отправить полное представление ресурса, даже если изменяется только часть полей. Сервер заменяет весь ресурс на предоставленные данные.

PATCH - используется для частичного обновления ресурса. Клиент отправляет только те поля, которые нужно изменить. Сервер обновляет только указанные поля, оставляя остальные без изменений.

3. Какие коды состояния HTTP наиболее часто используются в REST API?

- 200 OK - успешный запрос
- 201 Created - ресурс создан
- 400 Bad Request - неверный запрос (ошибка валидации)
- 404 Not Found - ресурс не найден
- 500 Internal Server Error - внутренняя ошибка сервера
- 204 No Content - успешный запрос без возвращаемого контента

4. Что такое ресурсо-ориентированный дизайн URL?

Это подход к проектированию URL, где каждый URL представляет собой конкретный ресурс:
- `/api/books` - коллекция книг
- `/api/books/1` - конкретная книга с ID=1
- `/api/books/1/authors` - авторы конкретной книги

URL должны быть существительными, а не глаголами, и отражать иерархию ресурсов.

5. Какие преимущества предоставляет пагинация в API?

- Уменьшение нагрузки на сервер - передаются только нужные данные
- Ускорение ответа - меньше данных для передачи и обработки
- Экономия трафика - особенно важно для мобильных приложений
- Удобство для клиента - постраничная навигация и предсказуемость
- Масштабируемость - система может работать с большими объемами данных

6. Как организовать валидацию данных в REST API?

1. Валидация на стороне клиента - для улучшения UX
2. Валидация на сервере - обязательная проверка всех входящих данных
3. Проверка обязательных полей - например, название и автор книги
4. Валидация форматов - email, ISBN, даты и т.д.
5. Проверка бизнес-логики - уникальность, существование связанных ресурсов
6. Возврат понятных ошибок - с указанием конкретных полей и причин

7. Какие подходы к обработке ошибок в REST API вы знаете?

- Единый формат ответов - одинаковый структурированный формат для всех ошибок
- Соответствующие HTTP-статусы - правильное использование кодов состояния
- Детальные сообщения - понятные описания ошибок для разработчиков
- Логирование - запись ошибок для последующего анализа
- Глобальный обработчик ошибок - централизованная обработка исключений
- Валидационные ошибки - отдельная структура для ошибок валидации

8. В чем разница между query parameters и path parameters?

Path parameters:
- Часть URL пути: `/api/books/1`
- Используются для идентификации конкретного ресурса
- Обязательные для маршрута
- Пример: ID книги, пользователя

Query parameters:
- Добавляются после `?` в URL: `/api/books?author=Толстой&page=1`
- Используются для фильтрации, сортировки, пагинации
- Необязательные параметры
- Пример: параметры фильтрации, номер страницы, лимит

Path parameters определяют "что", а query parameters - "как" отобразить или фильтровать.
